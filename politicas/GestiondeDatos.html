Políticas

Política de Gestión de Datos

Objetivo

Definir la forma en la que se almacenan los distintos datos así como la forma de acceder a ellos durante la fase de producción del proyecto,

  • 1: Almacenamiento de datos en general

    • 1.1: Tener conicimeito acerca de la Ley Federal de Datos Personales en Posesión de Particulares. Para saber más puede ver la siguiente presentación.
    • 1.2: Familiarizarse con el organigrama de Impulse donde se detalla cuáles son los roles existentes y quien los desempeña.
    • 1.3: Redactar un acuerdo de confidencialidad que deberá de ser firmado por todos los miembros dei equipo que participará en el proyecto o de lo contrario no podrá participar en el mismo.
    • 1.4: El cliente debe de designar a una persona o departamento que será el encargado de resolver las peticiones que los clientes puedan llegar a tener.
    • 1.5: El cliente debe de buscar asesoría legal para crear un Aviso de Privacidad donde se detalle:
      • 1.5.1: Quién recopila los datos
      • 1.5.2: Qué datos recopila (sensibles o no)
      • 1.5.3: Con qué finalidad los recopila
      • 1.5.4: Cómo limitar su uso o divulgación
      • 1.5.5: Cómo revocar su uso
      • 1.5.6: Cuál es el procedimiento que tiene el titular para ejercer sus derechos de acceso, rectificación, corrección y oposición (mejor conocidos como Derechos ARCO)
      • 1.5.7: La forma en la que se comunican cambios al Aviso de Privacidad la aceptación o negativa para autorizar la transferencia de datos a terceros.
    • 1.6: En caso de ser necesario, el cliente también debe crear un Aviso de Términos de Uso que debe ser elaborada con asesoría legal.
    • 1.7: En caso de que se almacenen datos de menores de edad, se debe de cumplir con todos los estatutos establecidos por la Ley de Protección de Datos de los Menores de Edad.

  • 2: API Keys

    En caso de que se manejen llaves para utilizar la API de algún servicio seguir los siguientes puntos para asegurar su integridad:

    • 2.1: Evitar que la llave aparezca de cualquier manera en el repositorio de GitHub.
    • 2.2: Evitar implementar de manera directa en el código las API keys.
    • 2.3: Las API Keys se guardan en archivos de texto dentro de una carpeta del proyecto llamada API’s a la cual únicamente tienen acceso los miembros del departamento.
    • 2.4: Si se necesita utilizar la API Key durante la implementación de una User Story se debe descargar el archivo correspondiente a la API que se desea utiliza. Para obtener la llave se lee el archivo txt y se almacena el valor en una variable local para ser utilizada. Este archivo txt debe ser incluido en el git ignore del proyecto para evitar que se suba al repositorio.

  • 3: Acceso a la base de datos.

    Existen distintos puntos a considerar dependiendo en que instancia se quiera acceder a la base de datos:

      Base da datos local:

      • 3.1: Utilizar contraseñas fáciles de recordar ya que facilita el proceso de desarrollo.
      • 3.2: En caso de crear distintos roles, almacenar en una carpeta las contraseñas que utilizan para acceder al sistema en una carpeta en Drive con el nombre de “Credenciales locales”. Sólo los miembros del proyecto deben tener acceso a dicha carpeta.
      • 3.3: Todos los miembros del equipo deben utilizar estas contraseñas provisionales durante el desarrollo del proyecto.

      Base de datos en producción:

      • 3.4: Cambiar todas las contraseñas que se utilizaron durante la fase de producción por unas que cumplan con parámetros de seguridad.
      • 3.5: Almacenar dichas contraseñas dentro de una carpeta en drive llamada “Credenciales de producción”. Sólo los Architecture Owners deben tener acceso a esta carpeta y otorgan el acceso cuando se realiza un cambio de rol.
      • 3.6: Encriptar las contraseñas de los usuarios que utilizan el sistema en caso de que el framework en el que se desarrolla no lo haga de manera automática.

  • 4: Credenciales del servidor

    En el caso de las credenciales para acceder al servidor en producción:

    • 4.1: Guardar las credenciales de acceso al servidor tanto de producción como de prueba en la carpeta del proyecto “Credenciales de producción”.

  • 5: Manipulación y acceso a la información

    Si el sistema manipula información confidencial del cliente se deben seguir los siguientes pasos para asegurar su protección:

    • 5.1: Todos los archivos que contengan información confidencial del cliente (archivos .word, .xslx, .png, .txt, etc…) deben de ser almacenados en una carpeta con el nombre del cliente dentro de la carpeta del proyecto.
    • 5.2: Cualquier solicitud de modificar o borrar la información confidencial del cliente debe ser comunicada a uno de los Product Owners.
    • 5.3: Cualquier solicitud de acceder a la información almacenada en el sistema por un tercero debe de tener la aprobación del cliente y del departamento de Impulse.
    • 5.4: Los miembros del equipo deben eliminar todos los archivos almacenados de manera local que contengan información confidencial del cliente al término del desarrollo del proyecto.
